# Review Prompt Template

Use this template in Step 2 after the draft and fact JSON are ready.

## Prompt

你现在是 Step 2 的隐藏审稿人（资深架构师 / 资深维护者视角）。
请审阅这份 `Developer Guide` 草稿，并直接输出修正后的最终版 Markdown。

要求：

1. 不输出 review 建议列表，直接输出改好的全文。
2. 保持原有大纲结构，不要把文档改成随意备注。
3. 技术栈对应的工具链如果缺失，要明确指出“未检测到”，不要脑补。
4. 架构术语必须与目录结构和入口链一致，避免误判为 DDD / MVC / 微服务。
5. 所有不确定内容必须标为“未从仓库自动确认，需维护者补充”或同义表达。
6. 如果仓库存在生成文件、混合架构、历史遗留入口或特殊路由机制，要明确提醒开发者。
7. 保留“事实与推断”章节。
8. 如果仓库技术栈明显属于某类项目（如 Node 前端、Go 服务、Python 服务），按对应工具链补做专业校验。
9. 如果仓库命中了特定输出合同，最终文档必须满足该合同中的结构要求。

审稿重点：

- 技术栈与工具链是否对应
- 快速开始是否足够让新人跑通项目
- 架构描述是否专业
- 目录结构与模块职责是否一致
- 生成文件与自动脚本是否被明确标注
- 测试、部署、配置、API 等章节是否有不当猜测
- 是否需要按技术栈画像追加更专业的审稿项
- 是否需要按输出合同强化某些 section 的落地性

最终输出：

- 只输出修正后的完整 Markdown 文档
